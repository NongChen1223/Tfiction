一、项目定位

项目名称：TFiction

定位：
- PC 平台本地化阅读器，主打小说阅读，漫画作为后续扩展方向
- 优先支持 Windows / macOS
- 核心目标不是“大而全”，而是“小而美”
- 产品价值集中在两个点：
  1. 阅读过程足够顺手、足够安静、足够稳定
  2. 摸鱼场景下足够隐蔽、足够快、足够少打扰

产品原则：
- 本地优先：用户数据、阅读进度、书架信息优先本地存储
- 轻量优先：不堆复杂系统，不为了“看起来功能多”牺牲速度和清晰度
- 细节优先：小功能要做完整，交互要自然，不允许半成品体验
- 聚焦优先：先把小说阅读和 BOSS 模式做到明显好用，再扩漫画、导出、统计等功能

当前阶段产品结论：
- 当前主产品形态应定义为：摸鱼导向的本地 TXT 小说阅读器
- EPUB / PDF / 漫画 / 书签 / 导出等属于后续阶段，不应在当前阶段抢主线

三、核心功能与交互设计

1. 书架设计

1.1 书架的角色
- 书架不是资源管理器替代品
- 书架只做三件事：快速找到书、快速继续读、快速进入摸鱼状态
- 任何会让书架变复杂的能力，都必须重新审视是否真的有必要

1.2 当前推荐交互
- 顶部保留：
  - 搜索框
  - 视图切换
  - 排序
  - 导入按钮
- 左侧保留：
  - 全部
  - 最近阅读
  - 设置入口
- 卡片保留：
  - 标题
  - 作者
  - 进度
  - 最后阅读时间
  - 继续阅读 / 目录内阅读

1.3 后续优化原则
- 不引入复杂资料库概念
- 不做太重的标签系统
- 优先补“最近阅读质量”和“继续阅读命中率”

建议优先补的小功能：
- P0 显示当前阅读中的书
- P0 更清晰的目录书籍状态
- P0 导入失败时给明确错误原因
- P1 空书架引导优化
- P1 最近阅读排序可切换

2. 小说阅读器设计

2.1 阅读器的目标
- 默认进入后不打扰阅读
- 常用动作一步可达
- 搜索和目录切换足够快
- 阅读状态恢复稳定，不能丢进度

2.2 当前已完成交互
- 左侧目录：展示章节、当前章节高亮、点击跳转
- 右侧搜索：输入关键字、查看结果、点击跳转
- 中间内容区：按设置即时应用样式
- 底部进度：显示总进度、章节位置、当前章节字数

2.3 需要继续强化的交互细节
- 顶部工具栏在普通阅读态也可以更轻量，不要太抢视线
- 搜索结果应补“当前结果高亮编号”和“上一条 / 下一条”
- 章节切换时需要更柔和的过渡，不要突兀跳变
- 长章节滚动保存可以进一步做节流和边界优化

建议优先补的小功能：
- P0 当前章节搜索
- P0 搜索结果上下跳转
- P0 阅读页返回书架时保留当前书架滚动位置
- P1 自动滚动
- P1 章节内小地图 / 进度刻度

3. BOSS 模式设计

3.1 BOSS 模式是项目核心卖点
- 它不应该只是“透明一点”
- 它应该像一个完整的阅读子模式
- 目标是：切得快、藏得稳、唤得回、调得顺

3.2 当前已完成交互
- 进入方式：
  - 阅读页按钮
  - 页面级快捷键
- 模式区分：
  - 基础隐身：保留内容，仅弱化控件
  - 完全隐身：内容和控件一起降到极低可见度
- 唤出逻辑：
  - 鼠标进入后显示
  - 鼠标移出后按设定延迟隐藏
- 右键面板：
  - 切换隐身模式
  - 调整透明度
  - 调整唤出延迟
  - 调整隐藏延迟
  - 锁定常显
  - 立即隐藏
  - 退出模式

3.3 BOSS 模式理想交互标准
- 进入动作：< 150ms 有明显反馈
- 内容显隐：平滑，不突兀，不闪烁
- 控件行为：只在需要时出现，不常驻干扰
- 快速隐藏：必须可靠，不应依赖多个条件判断
- 参数调整：即时生效，无确认按钮

3.4 后续必须继续做的方向
- P0 全局快捷键
- P0 托盘菜单切换
- P0 失焦自动降低暴露度
- P1 伪装层模式：
  - 空白文档风格
  - 代码编辑器风格
  - 表格工作风格
- P1 自动滚动和 BOSS 模式联动

4. 快捷键系统设计

4.1 当前状态
- 已支持设置页录制式自定义
- 已支持重复快捷键冲突检测
- 已支持恢复默认配置
- 已接入阅读页实际逻辑

4.2 当前已接入动作
- 切换 BOSS 模式
- 快速隐藏
- 打开搜索
- 上一章
- 下一章
- 返回书架

4.3 后续方向
- 页面级快捷键与全局快捷键分层
- 为不同模式提供不同快捷键表
- 增加用户可理解的冲突提示

四、后续优化规划

优化优先级原则：
- 先补“高频小功能”
- 再补“摸鱼关键体验”
- 最后才做“看起来很大”的功能

1. P0：接下来最值得做
- 全局老板键
- 系统托盘控制
- 自动滚动
- 搜索结果上下切换
- 当前章节内搜索
- 失焦自动隐匿
- 导入失败 / 不支持格式的明确提示

2. P1：第二阶段优化
- EPUB 支持
- PDF 支持
- 阅读页更轻量的工具栏
- 书架最近阅读体验优化
- BOSS 模式伪装层
- 章节内书签

3. P2：明确暂缓
- 漫画阅读器完整链路
- 导出系统
- 阅读统计分析
- 云同步
- 多端同步

五、设计与体验要求

1. 风格要求
- 整体维持当前风格，不做大改
- 保持轻量、安静、偏桌面工具的气质
- 不要花哨，不要堆重装饰

2. 动效要求
- 所有过渡以 150ms - 300ms 为主
- BOSS 模式显隐允许稍慢一点，但不能拖泥带水
- 避免无意义动画

3. 交互要求
- 常用操作必须一步到位
- 设置改动应即时生效
- 阅读状态必须可恢复
- 失败反馈必须明确

4. 开发要求
- 复用优先，不把逻辑散落在页面里
- 配置、状态、交互规则要可复用
- 注释只写关键原因，不写废话

六、项目结论

TFiction 的正确路线不是做成“什么都能读的超大阅读器”，而是做成：

一个专注小说阅读、对摸鱼场景非常友好、细节扎实、交互顺手的本地桌面阅读器。

后续所有功能取舍，都应该围绕这条主线：
- 是否让阅读更顺手
- 是否让摸鱼更安全
- 是否让产品更轻、更稳、更舒服
